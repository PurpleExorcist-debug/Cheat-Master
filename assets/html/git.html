GIT WORKFLOWS Y COMANDOS AVANZADOS

Comandos esenciales para el ciclo de vida, deshacer cambios, y reescribir la historia.

1. El Ciclo Básico (Staging, Commit, Push)

El flujo de trabajo diario: preparar los archivos, guardarlos localmente y subirlos al repositorio remoto.

Comandos:
# 1. Agrega todos los archivos modificados al "staging area"
git add .

# 2. Crea una instantánea (commit) de los archivos stageados
git commit -m "feat: Implementación inicial de la navegación principal"

# 3. Envía los commits locales a la rama remota (GitHub/GitLab)
git push origin nombre-de-tu-rama
Detalle:
  • **`git add .`**: Agrega *todos* los cambios. Para archivos específicos: `git add mi_archivo.js`.
  • **Mensajes (`-m`)**: Usa convenciones claras (ej. `feat:` para nuevas funcionalidades, `fix:` para correcciones).
2. Feature Branching (El Flujo de Trabajo Estándar)

Crea una rama separada para cada nueva funcionalidad o corrección, manteniendo la rama principal (`main` o `master`) estable.

Comandos:
# 1. Asegúrate de estar en la rama principal y actualízala
git checkout main
git pull origin main

# 2. Crea y cambia a la nueva rama
git checkout -b feature/nueva-tarjeta-producto

# (Trabaja y haz commits aquí)

# 3. Vuelve a main y combina (merge) tu trabajo
git checkout main
git merge feature/nueva-tarjeta-producto
git push origin main
Regla de Oro:
  • **Merge vs Rebase**: Para ramas de funcionalidad cortas, el `merge` es simple. Para historiales limpios en ramas a largo plazo, considera el `rebase` (ver Truco 4).
3. Deshacer Cambios (Reset y Revert)

Hay dos formas principales de volver atrás: `reset` (elimina el historial) y `revert` (crea un nuevo commit que anula el anterior).

Comandos (¡Úsalos con precaución!):
# Revertir un commit que ya fue enviado (¡Seguro!)
# Crea un nuevo commit que deshace los cambios del commit original
git revert <hash-del-commit-a-deshacer>

# Resetear al commit anterior (Elimina el historial local)
# --soft: Mantiene los cambios en el Staging Area
git reset --soft HEAD^

# --hard: ELIMINA PERMANENTEMENTE los cambios de los archivos
git reset --hard HEAD^
Cuándo usar cuál:
  • **`git revert`**: Para commits que **ya subiste** (públicos). Es el método más seguro.
  • **`git reset --hard`**: Solo para cambios **locales** que sabes que nadie más tiene.
4. Stashing (Guardar Trabajo Incompleto)

Necesitas cambiar de rama urgentemente, pero tu trabajo aún no está listo para un commit. `stash` guarda tus cambios y limpia tu directorio de trabajo.

Comandos:
# 1. Guarda los cambios actuales (Archivos modificados y stageados)
git stash save "Mi trabajo en el formulario de contacto"

# 2. Revisa la lista de stashes guardados
git stash list

# 3. Recupera y elimina el último stash (El trabajo vuelve a tu código)
git stash pop

# Opcional: Recupera sin eliminar el stash (úselo en otra rama)
git stash apply stash@{1}
Detalle:
  • **`pop`**: Lo aplica y lo elimina de la lista (es lo más común).
  • **Archivos Nuevos**: Para que `stash` funcione con archivos nuevos (untracked), debes hacer `git add .` primero.
5. Rebase (Historias Lineales y Limpias)

Mueve (o reescribe) toda la serie de commits de una rama y la coloca en la punta de otra. Esto evita los "merge commits" y mantiene un historial limpio y lineal.

Comandos:
# Estando en tu_rama, rebasea sobre la rama main
git fetch origin
git rebase origin/main

# Resuelve los conflictos si los hay, luego:
git add .
git rebase --continue

# Para forzar la subida del historial reescrito (¡Cuidado!)
git push --force-with-lease
Cuidado:
  • **No Rebasees Commits Públicos**: Solo rebasea el trabajo que no has compartido con nadie más. Rebasar commits públicos puede generar grandes conflictos a tus compañeros.
6. Squash Interactivo (Unir Commits)

Antes de un `merge` o `push` final, usa el rebase interactivo (`-i`) para combinar (squash) muchos commits pequeños y desordenados en un único commit limpio.

Comandos:
# 1. Abre el editor para los últimos 4 commits (HEAD~)
git rebase -i HEAD~4

# 2. En el editor que se abre, cambia 'pick' por 'squash' o 'fixup'
# pick <hash-1> Primer commit (Mantenlo como pick)
# squash <hash-2>
# fixup <hash-3>
# squash <hash-4>

# 3. Git te pedirá un nuevo mensaje para el commit combinado
Acciones en el Editor:
  • **`pick`**: Mantiene el commit.
  • **`squash`**: Combina el commit con el anterior (`pick`), pero te permite editar el mensaje final.
  • **`fixup`**: Combina el commit con el anterior (`pick`), pero descarta el mensaje de este commit (más rápido).
